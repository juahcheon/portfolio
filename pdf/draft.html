01
발단

운영은 AWS EC2, 테스트와 Admin은 각각 다른 Netlify로 배포 플랫폼이 세 곳이었습니다. 플랫폼이 달라 환경변수를 한 곳에서 관리할 수 없어, 처음엔 배포할 때마다 .env를 직접 수정하며 사용하고 있었습니다.

문제
사항

운영·Admin·테스트·로컬 네 환경을 오갈 때마다 .env를 수동으로 수정하다 보니 실수가 잦았고, 소셜 로그인 리다이렉트 URI도 환경마다 달라 매번 함께 변경해야 했습니다.

해결

런타임에 호스트명을 확인해 현재 환경을 자동으로 구분하는 유틸 함수를 작성했습니다. 배포 도메인이면 운영 API, 그 외엔 테스트 API를 자동으로 연결하고 리다이렉트 URI도 동일하게 처리해 .env를 수동으로 수정할 일이 없어졌습니다.

02
발단

이전 프로젝트에서는 배포할 때마다 SSH로 서버에 직접 접속해 명령어를 실행했습니다. DS Helper에서 운영 서버를 EC2로 이전하면서 GitHub Actions를 활용한 자동 배포를 도입했습니다.

문제
사항

deploy.yml을 온라인 예제에서 복붙했더니 파싱 오류가 계속 발생했고, 원인 파악에만 하루가 걸렸습니다. 오류를 해결하고 나니 이번엔 설정한 도메인으로 접속이 되지 않았습니다.

해결

오류 원인은 복붙 과정에서 섞여 들어온 invisible character였고, 직접 손으로 타이핑해 해결했습니다. 도메인 문제는 백엔드 개발자와 새벽 3시까지 함께 디버깅해 당일 배포를 완료했고, 이후 .env도 GitHub Secrets에서 자동 생성되도록 개선해 서버에 직접 접속할 일이 없어졌습니다.

발단

이전 Baro 프로젝트에서 사수 없이 혼자 개발하며, Zustand 하나로 서버 데이터까지 전부 관리하고 있었습니다.

문제
사항

Detail 페이지에서 새로고침하면 Zustand에 저장된 데이터가 초기화됐고, 정치인 정보처럼 대용량 데이터를 매번 재요청하다 보니 로딩이 느리다는 피드백도 받았습니다.

해결

01
발단

판결 결과 조회 API에서 DB 쿼리가 3번 발생했고, 전체 API 라우트에서 getServerSession을 사용해 인증 처리 비용이 높았습니다.

문제
사항

페이지 이동 시마다 불필요한 재요청이 발생했고, 판결 생성 후 window.location.reload()로 전체 페이지를 리로드하고 있었습니다.

해결

쿼리를 1번으로 통합하고, getServerSession을 경량 헬퍼 getRequestUserId로 교체했습니다. TanStack Query에 staleTime을 적용해 재요청을 줄이고, reload() 대신 invalidateQueries로 교체했습니다. 최대 9초, 평균 2~5초였던 API 응답 시간을 1초 이하로 개선했습니다.

02
발단

비용 절감을 위해 gemini-2.0-flash-lite를 사용했으나 미지원 오류가 발생했습니다. 또한 프롬프트에 실명을 넣으면 AI 응답에 개인정보가 그대로 노출됐습니다.

문제
사항

모델을 교체한 후에도 메타 추출·모더레이션까지 동일 모델을 사용해 비용이 높았고, 갈등 유형 선택 시 포괄적 유형만 반복 선택되는 정확도 문제도 있었습니다.

해결

판결은 gemini-2.5-flash, 경량 작업은 lite 모델로 분리했습니다. 프롬프트 내부는 A/B 레이블로 고정하고 표시 시 실명으로 치환했으며, 포괄적 유형을 최후 수단으로 제한해 정확도를 높였습니다.

03
발단

NextAuth 미들웨어가 페이지 라우트뿐 아니라 /api/* 라우트에도 적용되어 있었습니다.

문제
사항

배포 환경에서 인증되지 않은 API 요청이 미들웨어에 의해 307로 리다이렉트되어, 클라이언트에서 JSON 대신 HTML이 반환되는 문제가 발생했습니다.

해결

API 라우트는 getRequestUserId로 자체 인증을 처리하므로 미들웨어 매처에서 api/*를 전체 제외했습니다.
